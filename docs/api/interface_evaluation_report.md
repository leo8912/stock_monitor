# 数据接口评估报告：mootdx vs easyquotation

## 1. 核心差异对比

| 维度 | easyquotation (当前方案) | mootdx (提议方案) |
| :--- | :--- | :--- |
| **数据源** | 新浪/腾讯/HKEX Web API | 通达信行情服务器 (L1 协议) |
| **成交细节** | **无**。仅提供 5 档盘口快照。 | **完整**。支持获取当日每一笔成交明细。 |
| **性能** | HTTP 轮询，数据延迟 1-3 秒。 | TCP/IP 连接，延迟更低。 |
| **北交所/港股** | 仅支持部分（新浪源限制）。 | 全面支持（包含指数、港股、北交所）。 |
| **反爬风险** | 依靠模拟浏览器 Header，有封禁风险。 | 基于行业标准协议，相对更稳健（通达信客户端协议）。 |

## 2. 优劣势分析

### easyquotation (优势)
- **极简**: 无重度依赖，适合轻量级快照抓取。
- **项目现状**: 已经与现有的 `StockDataFetcher` 紧密集成，稳定性已在当前逻辑中验证。

### mootdx (优势)
- **数据深度**: 能实现“剔除100手小单”这种基于流向统计的复杂功能。
- **扩展性**: 支持 F10 财务数据、板块信息等更丰富的数据维度。
- **稳定性**: 直接对接行情服务器，不容易受网页版 API 变动的影响。

## 3. 评估建议：是否需要全面切换？

### 结论：建议采取“混合模式”，暂不全面切换。

**理由如下：**
1. **风险隔离**: `easyquotation` 对全量股票列表的基本行情获取（如价格、涨跌幅）非常成熟且并发压力小。
2. **平滑过渡**: 本项目现有代码结构中，`StockDataFetcher` 已经封装了基于 `easyquotation` 的批量拉取逻辑。全面切换需要重写数据转换模型（Processor）和获取链路（Fetcher），工作量较大且容易引入 Bug。
3. **按需引入**: 此次“资金流向”功能是针对**单股详情或自选股监控**的增强需求。

### 推荐路径：
- **核心行情**: 继续由 `easyquotation` 负责（获取关注列表的基础报价）。
- **增强功能**: 引入 `mootdx` 专门负责“资金流向统计”和“逐笔买卖分析”。

> 如果后续用户希望将软件升级为专业的“分笔分析助手”，届时再考虑将所有行情接口统一收拢到 `mootdx`。
