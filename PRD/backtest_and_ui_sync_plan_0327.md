# 需求文档：量化深度回测与 UI 排序同步优化 (2026-03-27)

## 1. 业务背景与问题定义
在近期性能优化过程中，发现系统存在以下待解决的回放问题：
1. **数据深度不足**：mootdx 单次请求限制在 800 根左右。之前设置的 15m (4000) 和 30m (2000) 根需求直接超出了服务器能力，导致回测引擎无法获取足够样本，从而显示“回测统计不全”。
2. **列表同步失效**：主界面刷新后，股票显示顺序由行情数据返回顺序决定（异步），导致即使在设置里排好了序，主界面显示也是乱的。
3. **状态残留**：确认保存设置后，异步刷新的“加载中”状态在特定条件下（如空数据或异常）无法自动消失，底部行显示“数据加载中”。

## 2. 核心功能需求
### A. K 线分段拉取 (Chunked Fetching)
- **方案**：在 `QuantEngine` 中实现 `fetch_bars_chunked`。
- **机制**：循环递增 `start` 参数（每次拉取 800 根），将结果拼接为 4000 根，最后按 `datetime` 升序排列。
- **目标**：彻底解决深度回测失败的问题。

### B. UI 强制排序与顺序同步
- **方案**：在 `MainWindow` 的数据接收环节增加“强制重排”步骤。
- **机制**：根据 `self.current_user_stocks` 数组定义的顺序，对返回的 `stocks_data` 进行重排序。
- **占位逻辑**：在触发 `refresh_now` 时，立即用当前顺序的“加载中”占位行覆盖旧数据。

### C. 加载标签自动收敛
- **方案**：增强 `loading_label` 生命周期管理。
- **机制**：增加超时逻辑，不论后端成功与否，必须保证 UI 遮罩层在 10s 内关闭。

## 3. 待变更代码范围

- `stock_monitor/core/backtest_engine.py`: 适配深度参数。
- `stock_monitor/ui/main_window.py`: 排序同步与 UI 占位逻辑。
